## Project Overview

JXNU选课PLUS — a single-page React application for browsing Jiangxi Normal University (江西师范大学) course catalog. The UI is entirely in Chinese (zh-CN). Course data is static (build-time Python pipeline); teacher ratings live in Cloudflare D1 via Pages Functions.

## Commands

- `npm run dev` — Vite dev server (port 5173, `server.host: true` so the LAN IP works)
- `npm run build` — `tsc -b` then `vite build`
- `npm run lint` — ESLint (flat config)
- `npm run preview` — preview prod build
- `npx tsc --noEmit` — type-check without emitting (used as quick gate after edits)
- `python build_data.py` — regenerate `public/courses.json` from the three raw university JSON sources

## Architecture

**Data pipeline** (`build_data.py`) — merges two university JSON exports into one denormalized `public/courses.json`:
1. `JXNU课程数据_*.json` — course catalog (id, name, credits, dept, teachers, ...)
2. `jxnu_jxjh_v5_*.json` — 培养方案 (training plan); per-`(年级, 专业, 方向)` entries with `课程性质` and a `学位课程` flag

(An older `JXNU开课安排_*.json` schedule file existed to infer a "专业课" tag, but that tag has been retired in favor of more granular nature tags from the training plan.)

Each course gets a `plans: CoursePlan[]` array aggregating every training-plan appearance, plus `isDegreeCourse: boolean` (any plan has `学位课程` non-empty). Course-nature values from training plans become additional `tags`; `公共必修` is normalized to `公共必修课` via `NATURE_NORMALIZE` to align with the catalog's terminology.

**Client-side SPA**: React app fetches `courses.json` on mount and does all filtering/sorting/pagination client-side. No backend for course data.

**Rating system** — Cloudflare Pages Functions in `functions/api/`, D1 binding `DB` (db `jxnu-ratings`, schema in `d1_schema.sql`). The single `ratings` table is uniquely keyed by `(course_id, teacher_id, voter_id)`. Endpoints:
- `GET /api/ratings?courseId=xxx` — teacher averages for one course
- `GET /api/ratings/all` — all ratings (bulk-load for table view)
- `GET /api/ratings/check?teacherId=xxx&voterId=xxx&courseId=xxx` — has-this-user-rated check
- `POST /api/ratings` — submit (upsert) `{ courseId, teacherId, rating, voterId }`
- `DELETE /api/ratings` — remove `{ courseId, teacherId, voterId }`

Voter dedup uses a localStorage UUID v4 generated by `src/lib/voter.ts`. **Do not reintroduce `crypto.randomUUID()` directly** — it's only available in secure contexts (HTTPS/localhost) and crashed the module load over HTTP LAN access. The current implementation falls back through `crypto.randomUUID` → `crypto.getRandomValues` → `Math.random`.

**State / data flow**:
- `src/lib/ratingsStore.ts` — module-level subscription store (real + optimistic maps) shared across components, with optimistic update + rollback pattern. Custom hooks in `src/hooks/useRatings.ts` (`useRatings`, `useAllRatings`) subscribe via `useSyncExternalStore`-style pattern.
- `useCourseData` — fetches `/courses.json`, exposes `allPlans` (`年级-专业` keys for the plan selector) and `courseTypes` (hardcoded list: 公选课/公共必修课/教师教育课程/专业主干/专业限选/专业任选/专业类基础/大学英语特色课) and `subTags = allTags - courseTypes`.
- `useCourseFilter` — tri-state filter cycle (default → include → exclude → default) on credits/dept/type/tag, with `sessionStorage` persistence.

**Routes** (`App.tsx`):
- `/` → `HomePage` (desktop 3-column / mobile single-column with slide-up overlay)
- `/course/:id` → `CourseDetailPage`

`CourseDetailPage` is intentionally a thin wrapper: it loads the course and delegates rendering to `<CourseDetail course={...} onClose={() => navigate("/")} />`. **Keep this delegation.** Earlier the page had its own duplicated detail UI that drifted out of sync (no 培养方案归属 section, etc.) — DO NOT reintroduce a parallel implementation. All detail UI changes go in `src/components/CourseDetail.tsx`.

`HomePage` uses `window.history.pushState` (not `navigate`) when opening the mobile overlay so React Router stays on `/` and the overlay state in `HomePage` is preserved. Only refresh / direct-link / external nav hits the `/course/:id` route.

## Search Index (`_search`)

Built in `build_data.py` `build_search()`. Includes: course id, course name, dept, teacher names, **teacher ids**, tags. Excludes: anything from `plans` (year, major, direction, nature) — those keywords are too broad and produce noise (e.g. searching "计算机" would otherwise match every course in the CS curriculum). If you add new fields to courses, decide deliberately whether they belong in the search index.

## StarRating Component

Renders fractional stars via **two stacked SVGs** (gray base + gold overlay clipped by width %). **Do not switch back to `<linearGradient id="...">`** — SVG `id`s are document-global; multiple `StarRating` instances on one page (which is the norm in the course list) collided and lost half-star fills on iOS Safari.

## Key Types (`src/types.ts`)

`Course` includes the static catalog fields plus `tags: string[]`, `teachers: Teacher[]`, `isDegreeCourse: boolean`, `plans: CoursePlan[]`, and a prebuilt `_search: string`. `CoursePlan = { year, major, direction, nature, isDegree, semester }`.

## Tech Stack

React 19 · TypeScript 6 · Vite 8 · Tailwind 4 (via `@tailwindcss/vite`, **no `tailwind.config.*`** — uses `@import "tailwindcss"` in `src/index.css`) · React Router 7 · ESLint 10 flat config · Cloudflare Pages Functions + D1.

## Data Update Workflow

1. Drop the new raw JSON files in the project root.
2. Update the relevant `*_FILE` constants at the top of `build_data.py` (`COURSE_FILE`, `SCHEDULE_FILE`, `TRAIN_PLAN_FILE`).
3. `python build_data.py` → writes `public/courses.json` and prints tag distribution.
4. Verify with a sample check, e.g.:
   ```bash
   python -c "import json; d=json.load(open('public/courses.json',encoding='utf-8')); print(len(d), sum(1 for c in d if c['plans']), sum(1 for c in d if c['isDegreeCourse']))"
   ```

## Branch Conventions

`test` is the staging branch (preferred for hotfixes before promoting to `main`). `main` is the deployed branch — Cloudflare Pages auto-deploys from it. D1 schema/API changes need to land in `wrangler.toml` / `functions/api/` / `d1_schema.sql`; verify there before merging if a change might touch ratings data.
